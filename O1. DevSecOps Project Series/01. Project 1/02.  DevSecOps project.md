## Overview

This project is designed to help you build a small application and place it inside a secure CI/CD pipeline.

The application itself is intentionally simple. The real focus is the delivery process around it, how code is tested, how secrets are detected, how vulnerabilities are scanned, how merges are controlled, and how deployments are approved.

This is a good home lab project if you want something practical for your GitHub portfolio that shows real DevSecOps thinking without needing a large cloud environment.

---

## What This Project Covers

By the end of this project, you will have a workflow that:

- runs tests on pull requests
- scans for leaked secrets
- scans the codebase for high and critical issues
- blocks risky code from reaching `main`
- requires approval before release
- builds and pushes a Docker image only after approval
- can optionally generate build provenance attestation

## Step 4. Secure the Repository Settings

Before adding workflows, configure the repository properly.

### Create the `production` environment

Go to:

`Repository` → `Settings` → `Environments` → `New environment`

Create:

```text
production
```

Inside that environment:

- enable required reviewers
- add yourself or a teammate
- restrict deployment branches to `main`
- only enable prevent self-review if another reviewer is available

### Protect the `main` branch

Go to:

`Settings` → `Branches`

Create a branch protection rule for `main` and enable:

- require a pull request before merging
- require approvals
- require status checks to pass before merging
- include administrators, optional but recommended
- restrict direct pushes if your plan allows it

### Set GitHub Actions permissions

Go to:

`Settings` → `Actions` → `General`

Recommended setting:

```text
Read repository contents
```

This keeps the default token permissions narrow until a workflow genuinely needs more.

## Step 5. Add the CI Workflow

Create the workflow directory:

```bash
mkdir -p .github/workflows
```

Create `.github/workflows/ci.yml`:

```yaml
name: CI Security Pipeline

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read

jobs:
  test-and-scan:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

      - name: Run Gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Run Trivy filesystem scan
        uses: aquasecurity/trivy-action@0.33.1
        with:
          scan-type: 'fs'
          scan-ref: '.'
          format: 'table'
          exit-code: '1'
          ignore-unfixed: true
          severity: 'CRITICAL,HIGH'

      - name: Upload Trivy report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: trivy-fs-report
          path: .
```

This workflow:

- runs on pull requests and pushes to `main`
- installs dependencies
- runs tests
- scans the repository for secrets
- scans the filesystem for high and critical findings

Commit and push it:

```bash
git add .github/workflows/ci.yml
git commit -m "Add CI security workflow"
git push
```

## Step 6. Add the Release Workflow

Create `.github/workflows/release.yml`:

```yaml
name: Release Secure Image

on:
  workflow_run:
    workflows: ["CI Security Pipeline"]
    types:
      - completed

permissions:
  contents: read
  packages: write
  id-token: write
  attestations: write

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  release:
    if: >
      github.event.workflow_run.conclusion == 'success' &&
      github.event.workflow_run.head_branch == 'main'
    runs-on: ubuntu-latest
    environment: production

    steps:
      - name: Checkout repository at tested commit
        uses: actions/checkout@v4
        with:
          ref: ${{ github.event.workflow_run.head_sha }}

      - name: Log in to GHCR
        run: echo "${{ secrets.GITHUB_TOKEN }}" | docker login ghcr.io -u "${{ github.actor }}" --password-stdin

      - name: Build Docker image
        run: |
          IMAGE_ID=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          IMAGE_ID=$(echo "$IMAGE_ID" | tr '[:upper:]' '[:lower:]')
          echo "IMAGE_ID=$IMAGE_ID" >> $GITHUB_ENV
          docker build -t $IMAGE_ID:${{ github.event.workflow_run.head_sha }} .
          docker tag $IMAGE_ID:${{ github.event.workflow_run.head_sha }} $IMAGE_ID:latest

      - name: Run Trivy image scan
        uses: aquasecurity/trivy-action@0.33.1
        with:
          image-ref: ${{ env.IMAGE_ID }}:${{ github.event.workflow_run.head_sha }}
          format: 'table'
          exit-code: '1'
          ignore-unfixed: true
          severity: 'CRITICAL,HIGH'

      - name: Push image
        run: |
          docker push $IMAGE_ID:${{ github.event.workflow_run.head_sha }}
          docker push $IMAGE_ID:latest

      - name: Get image digest
        id: image_digest
        run: |
          DIGEST=$(docker inspect --format='{{index .RepoDigests 0}}' $IMAGE_ID:${{ github.event.workflow_run.head_sha }} | awk -F'@' '{print $2}')
          echo "digest=$DIGEST" >> $GITHUB_OUTPUT

      - name: Generate build attestation
        uses: actions/attest-build-provenance@v3
        with:
          subject-name: ${{ env.IMAGE_ID }}
          subject-digest: ${{ steps.image_digest.outputs.digest }}
          push-to-registry: true
```

This workflow:

- waits for CI to pass
- runs only for `main`
- pauses for `production` approval
- builds and scans the image
- pushes the image to GitHub Container Registry
- generates provenance attestation if enabled

Commit and push it:

```bash
git add .github/workflows/release.yml
git commit -m "Add secure release workflow"
git push
```

## Step 8. Test the Security Controls

A good project should prove that its controls actually work.

### Secret leak test

Create a new branch:

```bash
git checkout -b feature/bad-secret-test
```

Create a file called `fake-secret.txt` with this fake value:

```text
AWS_SECRET_ACCESS_KEY=abcd1234abcd1234abcd1234abcd1234abcd1234
```

Push it and open a pull request.

Expected result:

- Gitleaks fails
- the pull request cannot move forward cleanly

Take a screenshot, then remove the file and clean up the branch.

### Vulnerable image test

Make the `Dockerfile` intentionally riskier by using an older base image or adding something known to trigger scanning issues.

Push that change in a test branch and open a pull request.

Expected result:

- Trivy flags the issue
- the workflow fails

Take a screenshot, then revert the change.

## Common Mistakes

### Direct pushes to `main`

Use branch protection.

### Deployment runs without approval

Make sure the release job uses the `production` environment and that required reviewers are configured.

### Secrets are stored in workflow files

Never hardcode real secrets in YAML files. Use GitHub Secrets.

### Workflow permissions are too broad

Start with read-only permissions and increase them only where necessary.

### Actions are pinned only to tags

This is okay for learning, but pinning to full commit SHAs is stronger later.

### Environment secrets are expected before approval

Environment secrets are only available after the protection rules for that environment are satisfied.
